# 0001 MVP

Status: Draft

## Goal

Build a single-container web scheduler that runs automation jobs on daily or
weekly schedules and persists config, user scripts, and logs through a mounted
data directory.

## Target Environment

- Synology Container Manager / Docker.
- One app container.
- Single operator.
- Persistent mount at `/data` by default.

## Initial Scope

- Daily schedule: run once per day at `HH:MM`.
- Weekly schedule: run once at `HH:MM` on selected weekdays.
- Bash job runtime.
- One user-provided script per job.
- Save-before-run and test-run flows for job scripts.
- Global environment and job-specific environment.
- Run logs and run index.
- Web UI after core logic and HTTP API are working.

## Non-Goals

- Multi-user auth.
- Monthly schedules.
- Interval schedules.
- Multiple executions per day for one schedule.
- External queues.
- Database-backed state.

## Persistent Layout

```text
/data
  config.json
  logs/
    index.jsonl
    runs/
  scripts/
    jobs/
```

## Config Direction

Use a human-editable config file. Start with JSON to avoid external runtime
dependencies; YAML can be added later if manual editing becomes painful.

The persisted config should contain:

- app version
- timezone
- global environment config
- jobs

Each job should contain:

- generated id
- name
- enabled
- job-specific environment config
- runtime config, including the app-managed script path
- schedule config

The normal web UI should not ask the user to type internal IDs or script file
paths. It should generate and manage those values when saving a job.

## UI Direction

The UI comes after core logic is unit-tested. It should support:

- list jobs
- create/edit/enable/disable jobs
- choose daily or weekly schedule
- edit script content, runtime, timeout, and environment names
- run a draft job as a test and show output before saving
- show next run time
- show recent runs and logs
